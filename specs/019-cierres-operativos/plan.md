# Implementation Plan: Cierres Operativos

**Branch**: `019-cierres-operativos` | **Date**: 2026-06-08 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/019-cierres-operativos/spec.md`

## Summary

Implements the day-closing module for Rochester Viandas. Executing a close atomically marks all confirmed-but-undelivered orders as `no_retirado`, calculates daily totals (deliveries, surplus sales, cash revenue), and persists a `CierreOperativo` record that blocks further operations on that day. The module exports `isDiaCerrado` for use by `EntregasService` and `VentasSobrantesService`.

## Technical Context

**Language/Version**: TypeScript / Node.js 18+ (project standard)

**Primary Dependencies**: NestJS, TypeORM, class-validator, PostgreSQL

**Storage**: PostgreSQL — table `cierres_operativos` with composite UNIQUE on `(tenant_id, fecha_operativa, sede_id, punto_retiro_id)` and index on `(tenant_id, fecha_operativa, sede_id)`

**Testing**: Jest (project standard)

**Target Platform**: Railway / AWS-compatible Linux server

**Project Type**: NestJS REST API — Type C module (purely operational, no CRUD base)

**Performance Goals**: `isDiaCerrado` resolves in < 5ms (indexed lookup). Cierre execution < 10s.

**Constraints**: All business errors via `AppError` + `ErrorCodes`. No `throw new Error()`. `SELECT FOR UPDATE` on pedidos during close. `GET /resumen-previo` declared before `GET /:id` in controller.

**Scale/Scope**: One close record per day/sede/punto_retiro per tenant. History queries up to 90 days.

## Phase 0: Research

All research complete. See [research.md](research.md).

Key decisions:
1. **QueryRunner with SELECT FOR UPDATE** for transactional cierre (consistent with ventas-sobrantes and entregas pattern)
2. **Only `CONFIRMADO_PAGO_ONLINE` and `CONFIRMADO_PAGO_PRESENCIAL`** marked as `no_retirado`
3. **`isDiaCerrado`** as a simple `count` query using the composite index — no transaction needed
4. **Recaudación** = `SUM(entrega_pedidos.importe_cobrado_caja)` + `SUM(ventas_sobrantes.importe_total)`
5. **`GET /resumen-previo` before `GET /:id`** in controller declaration order to avoid NestJS routing conflict
6. **Two-layer uniqueness**: explicit `findOne` check for descriptive error + DB UNIQUE constraint as race condition guard
7. **No circular dependency**: `CierresOperativosModule` reads entrega/ventas repos directly; those modules import `CierresOperativosModule` (not vice-versa at module level)

## Implementation Notes

### Entity highlights

```typescript
@Entity('cierres_operativos')
@Unique(['tenant_id', 'fecha_operativa', 'sede_id', 'punto_retiro_id'])
@Index('idx_cierre_tenant_fecha_sede', ['tenant_id', 'fecha_operativa', 'sede_id'])
// NO updated_at, NO deleted_at — immutable record
```

### ejecutarCierre transactional flow (QueryRunner)

```
1. findOne({ tenant, fecha, sede, punto }) → throw CIERRE_YA_EXISTE if found
2. QB pedidos WITH pessimistic_write
   WHERE estado IN [confirmado_pago_online, confirmado_pago_presencial]
     AND fecha_retiro = fecha AND sede_id AND punto_retiro_id AND tenant_id
     AND deleted_at IS NULL
3. pedido.estado_pedido = NO_RETIRADO → save each via qr.manager
4. COUNT entrega_pedidos (fecha::date, sede, punto, tenant)
5. SUM entrega_pedidos.importe_cobrado_caja (same filter)
6. COUNT ventas_sobrantes (fecha, sede, punto, tenant)
7. SUM ventas_sobrantes.importe_total (same filter)
8. CREATE + SAVE CierreOperativo with totals and fecha_cierre = new Date()
9. auditService.write('cierre.operativo.registrado') — same qr.manager
10. commitTransaction()
```

### isDiaCerrado (exported, used by entregas + ventas-sobrantes)

```typescript
async isDiaCerrado(fecha: string, sedeId: string, puntoRetiroId: string, tenantId: string): Promise<boolean> {
  const count = await this.cierreRepo.count({
    where: { tenant_id: tenantId, fecha_operativa: fecha, sede_id: sedeId, punto_retiro_id: puntoRetiroId }
  });
  return count > 0;
}
```
